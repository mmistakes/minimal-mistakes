# Blog Style

## 목적과 우선순위

- 이 문서는 `_posts` 아래 기술 블로그 글의 작성·개정 기준을 정의한다.
- 우선순위는 `검증 가능성 > 구조적 신뢰성 > 유지보수성 > 문체`다.
- 글을 멋있게 쓰는 것보다, 무엇이 확인된 사실이고 무엇이 해석인지 분리하는 것을 우선한다.

## 필수 front matter

- `title`: 검색 의도가 드러나는 구체적 제목
- `date`: 문서 발행일
- `lang`: `ko` 또는 `en`
- `translation_key`: KOR/ENG 연결 키
- `section`: `security`, `development`, `life` 중 하나
- `tags`: 배열 형태
- `description`: 검색 랜딩과 요약에 쓸 1문장 설명

필요 시 추가한다.

- `topic_key`: 같은 허브나 시리즈로 묶일 때
- `permalink`: 영어 미러 글에서 필수
- `sidebar.nav: "sections"`: `POSTS` 사이드바 유지가 필요할 때

## 필수 본문 구조

기술 글은 아래 헤더를 그대로 유지하는 것을 기본으로 한다. 헤더를 고정하면 리뷰와 후속 개정이 쉬워진다.

1. `## 요약`
2. `## 문서 정보`
3. `## 문제 정의`
4. `## 확인된 사실`
5. `## 직접 재현한 결과`
6. `## 해석 / 의견`
7. `## 한계와 예외`
8. `## 참고자료`

`## 문서 정보`에는 아래 항목을 빠짐없이 넣는다.

- `작성일`
- `검증 기준일`
- `문서 성격`: `tutorial`, `analysis`, `comparison`, `opinion` 중 하나 이상
- `테스트 환경`
- `테스트 버전`

실행 테스트가 없는 글도 빈칸으로 두지 않는다.

- 실행이 없으면 `테스트 환경: 실행 테스트 없음`처럼 이유를 적는다.
- 버전을 고정하지 않았다면 `테스트 버전: 미고정`처럼 적는다.
- 직접 재현이 없다면 `## 직접 재현한 결과`에 그 사실과 이유를 적는다.

## 사실, 실험, 의견 구분 규칙

- 공식 문서나 표준, 원저자 저장소에서 확인한 사실은 `공식 문서에 따르면`, `표준 문서 기준`, `원저자 저장소 기준`처럼 출처 성격이 드러나게 적는다.
- 직접 실행, 재현, 관찰한 내용은 `직접 확인한 결과`, `직접 재현한 결과`, `샘플 관찰`처럼 실험임을 드러내는 표현을 쓴다.
- 해석이나 권고는 `내 판단으로는`, `해석:`, `의견:`처럼 사실과 분리한다.
- 검증하지 못한 내용을 사실 섹션에 넣지 않는다. 필요한 경우 `확인하지 못했다`, `가능성이 있다`, `추가 검증이 필요하다`처럼 불확실성을 남긴다.

## 근거 연결 방식

- 핵심 주장은 같은 문단 안이나 바로 다음 줄에서 근거와 연결한다.
- 링크는 글 끝 참고자료에만 모아두지 말고, 주장 옆에도 배치한다.
- 출처 우선순위는 아래 순서를 기본으로 한다.

1. 공식 문서
2. 표준 문서
3. 원저자 저장소
4. 공식 발표 자료
5. 공신력 있는 기관 자료
6. 2차 자료

- 2차 자료를 써야 한다면 왜 1차 출처를 대신하는지 문맥상 드러나게 적는다.

## 최신성, 날짜, 버전 규칙

- 서비스 기능, 정책, 가격, UI, CLI 동작, 명령어, SDK 동작처럼 바뀌기 쉬운 내용은 반드시 `검증 기준일`을 적는다.
- `현재`, `요즘`, `최신` 같은 표현은 날짜 없이 쓰지 않는다.
- 버전 의존적인 주장에는 최소한 도구 이름, 운영체제 또는 실행 환경, 확인한 버전을 적는다.
- 재현 조건이 제한적이면 그 범위를 명시한다.

## 글 유형별 최소 요구사항

### 튜토리얼

- 재현 순서
- 필요한 선행 조건
- 실행 명령 또는 조작 단계
- 성공 조건
- 실패 가능 조건

### 비교 글

- 비교 기준
- 같은 조건인지 여부
- 제외한 항목
- 결론이 적용되지 않는 상황

### 분석 글

- 분석 대상 범위
- 사용한 자료나 샘플
- 직접 본 것과 해석한 것의 경계
- 관찰하지 못한 영역

## 한계와 예외 작성 규칙

- 결론이 통하지 않는 환경이 있으면 명시한다.
- 테스트하지 않은 운영체제, 버전, 설정, 반례 가능성을 숨기지 않는다.
- 단정형 표현을 썼다면 어떤 조건에서만 성립하는지 같이 적는다.

## 기존 글 개정 시 적용 방법

- 먼저 기존 문장을 `확인된 사실`, `직접 재현한 결과`, `해석 / 의견`으로 분류한다.
- 근거가 없는 단정문은 삭제하거나 의견으로 낮춘다.
- 날짜나 버전이 빠진 최신성 민감 문장은 `문서 정보`와 본문 근거에 기준일을 추가한다.
- 기존 KOR/ENG 쌍이 있다면 `translation_key`, 제목, 요약, 본문 구조 차이가 과도하지 않은지 같이 본다.

## 검토 체크리스트

- 필수 섹션 8개가 모두 있는가
- `문서 정보` 5개 항목이 모두 채워져 있는가
- 핵심 주장마다 근거가 연결되어 있는가
- 사실과 의견이 섞여 있지 않은가
- 직접 재현 결과에 환경과 버전이 있는가
- 튜토리얼/비교/분석 글에 한계와 예외가 있는가
- 최신성 민감 내용에 날짜가 있는가
- 검증하지 못한 내용에 불확실성이 표시되어 있는가
